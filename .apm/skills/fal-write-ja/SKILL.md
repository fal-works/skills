# 日本語

このスキルは、日本語で書く任意の出力に適用します。文書、コードコメント、コミットメッセージ、チャットの返信のいずれも対象であり、会話でも基準は緩めず、口語へ崩すこともしません。リポジトリ内の文章では fal-write-docs と併用します。

## 文体と表記

### 敬体・常体の選択

ユーザーから指定がない限り、チャットの返信では「です・ます」調、文書では「だ・である」調で書きます。
ソースコードのコメントは既存のコメントに合わせ、特に定まっていなければ「だ・である」調とします。

## 日本語としての自然さ

LLMの生成する日本語は、文法が正しくても、語の選択と言い回しの不自然さが高い密度で現れます。失敗は語や言い回しを選ぶ時点で入り込み、書いた本人には自然に見えるため、規則を読んでいても生成の時点では止められないことが多いです。したがって、対策の中心は書き上げた後の推敲に置きます。

このスキルの規則は、二つの時点で働きます。まず生成の時点で、LLMが表現を選ぶデフォルトを置き換えて失敗の量を減らします。次に推敲の時点で、判定の基準になります。まとまった量を書いたときは、書き上げてから fal-improve-ja の手順で必ず推敲します。目標は、日本語話者がその場面で実際に書く形のみでできた文章です。

失敗の原因は大きく2つあります。第一は、英語の影響です。LLMは英語から翻訳して日本語を書くわけではありませんが、生成の土台となるパターンは、語義の対応づけから言い回し、構文、文章の展開に至るまで、英語圏の文章に強く影響されており、日本語は翻訳調へ傾きます。第二は、短く言い切ろうとする圧力です。文字数を切り詰めようとすると、なじみの薄い漢語が選ばれ、複合語が作られ、助詞が削られます。

日本語の技術文書では、説明的な言い回しこそが標準であり、冗長ではありません。fal-write-docs の簡潔さの原則（§1 Write only what earns its place, §6 One role per paragraph or list, §7 One thought per sentence）は何を書くかを決めるものであり、どう言い表すかの水準で簡潔さと自然さが衝突したら、自然さを取ります。

このスキルの中心となる §1〜§3 は、fal-write-docs §4 (Use words that resolve) の日本語への拡張であり、日本語では、語の裏付けとして、指示対象だけでなく実際の用法まで求めます。以下、§1 が全体を貫く判定基準を定めます。§2 と §3 はこの二つの原因に語と言い回しの単位で対処し、§4〜§6 は同じ失敗が文の組み立て、修辞、記号に現れる形を扱います。§7 は適用の加減を定めます。

### 1. 語は意味ではなく用法で選ぶ

語が正しいのは、その語がその意味で、その種類の文章の中で実際に使われている場合だけです。意味が合うというだけで選ぶと、近い意味の語が場面ごとに分かれている場合（評価・評定・査定・考課）に、別の場面の語を選んでしまいます。実際に使われているかどうかは、その語がその意味で使われている文章を読んだ覚えがあるかで判定します。その場で例文を作れることは証拠になりません。意味が通る語なら例文は作れてしまうからです。読んだ覚えがない語は、実在しないとは限りませんが、覚えがないこと自体が、選ばない理由として十分です。説明的な言い回しに替えても失うものはありません。

- **場面**: 語の意味が合っていても、その語が実際に使われる場面（分野・対象・状況）が文脈と合わないことがあります。例: レビューの評価を「評定」と書いてしまう。
- **文体**: 話し言葉と学術文体のどちらへ傾いても、読者に合わなくなります。例: 方式の推奨を「推す」と書いてしまう。
- **組み合わせ**: 語も意味も合っていても、その組み合わせで実際に使われるとは限りません。動詞と主語・目的語の間でも同じです。例: 一覧が網羅的でないことを「リストは尽きない」と書いてしまう。
- **言語**: 日本語の文中の英単語は術語として読まれます。コード上の識別子、定着した専門用語、プロジェクトで定義済みの語のいずれかに当たる場合に限って使い、それ以外は日本語で書きます。例: 定着した術語ではない login flow は、「ログイン処理の流れ」と書きます。

### 2. 圧縮した複合表現を造らない

定着した語がない概念に出会ったとき、または短く言い切ろうとしたときの典型的な失敗は、複数の語を助詞や活用なしにつないで、定着した語であるかのような表現を作ってしまうことです。最も見抜きにくいのは漢語の連結です。漢語の連結は語形成としていつでも可能で、できた語は見た目では既存語と区別できません。同じ失敗は数詞・カタカナ語・英字を含む連結にも起きます。fal-write-docs §4 で unmarked coinage と呼ぶ失敗の日本語版です。

削られるのは文字数だけではありません。たとえば「決定論的なパース」を「決定論パース」と書いてしまうと「的な」とともに元の意味が失われますし、「3段階で移行する。各段階からロールバック可能」を「3段階ロールバック可能移行」と圧縮すると単語同士の関係が復元できなくなります。造語の類型ごとの解説と実例は、fal-improve-ja の用例集に記載しています。

連結を構成する語にも §1〜§3 はそのまま適用されます。「著述摩擦」の「摩擦」（手間の意味の friction の直訳）のように、単独の語なら気づける失敗も、複合語の中では見えにくくなります。

一方、語の連結そのものは日本語の通常の語形成であり、型として定着した連結は、その連結を読んだ覚えがなくても使えます。型は2つあります。接辞のように任意の語と結びつく要素との連結（「未検証」「検証済み」「既存機能」「要確認」）と、後項が種類を表す語で、前項がその内容を限定する2語の連結（「検証手順」「比較実験」）です。どちらの型でも、省いた語を補って戻したときに、戻し方が一通りに決まることが条件です。補う語は、助詞（「検証手順」→「検証の手順」）のこともあれば、「未」「済み」のような接辞をほどいた言い方（「未検証」→「まだ検証していない」）のこともあります。

型の定着は、その型でできた既存の語が複数あるかで確かめます。「検証手順」は「作業手順」「導入手順」が裏付けます。向きが違えば別の型であり、「年跨ぎ」があっても「跨ぎ参照」は作れません。この型に当たる複合語まで「検証の手順」のように言い直すことを徹底すると、かえって日本語話者が書く形から離れます。型ごとの実例は、fal-improve-ja の用例集に記載しています。

定着した語があればそれを使います。なければ造語せず、説明的な言い回しで書くか、原語のまま書きます（single source of truth なら「唯一の情報源」か「SSoT」）。語を補った言い回しは、長くなっても正しい日本語です。確信のない連結は、外れたとき、既存語と見分けの付かない造語になって残ります。迷ったときは言い回しを採ります。

一つの文書で繰り返し使う複合表現は、初出で定義すれば使えますが、控えめにします。術語を新しく作って文書を越えて広めるのは命名という設計行為なので、黙って導入せずユーザーに提案します（fal-write-docs §4 Use words that resolve）。

### 3. 英語の語義や言い回しを持ち込まない

不自然な日本語の背後には、たいてい自然な英語の表現があります。語の意味範囲は言語間で一致しないため、英語の語義を対応する日本語の語へそのまま持ち込むと、日本語にはない用法でその語を使うことになります。同じことは名詞や形容に使う語（「正直な比較」= honest）だけでなく、動詞（「〜に解決する」= resolve）や、慣用句・比喩・修辞の表現全体（「無料で付いてくる」= for free）でも起きます。

書くときは、頭にある英語に対応する語を探すのではなく、この場面で日本語話者なら何と言うかを考えて、そちらを書きます。直すときも同じで、直訳された語を別の語に置き換えるのではなく、元の英語が文脈で意味していた内容を取り出して、それを日本語で言い直します。

英語で書かれたスキルや指示を読みながら日本語を書くと、その言い回しの直訳も紛れ込みます。スキルや文書が定義している英語の術語は、その場で訳さず、原語のまま使います（§1 の定義済みの語にあたります）。その場で作った訳語は、定義とのつながりを失った造語になります。

例: read as を「〜と読む」、prose を「散文」と直訳してしまう。

### 4. 文と句の組み立て

語がすべて正しくても、文や句の形で失敗が起きます。失敗が入り込む経路は語の場合と同じで、英語の構文の直訳か、圧縮です。

- **無生物主語と人の述語**: 人の行為を表す述語（沈黙する・名指す・語る）を無生物主語と組み合わせると、翻訳調になります。無生物主語そのものは日本語でも普通に使います（「規則が定める」「変更が影響する」）ので、避けるのは述語との組み合わせであり、判定は §1 と同じです。例: エラーを出さずに失敗することを「エラーが沈黙する」と書いてしまう。
- **助詞の省略**: 助詞は無駄ではありません。その場の文の内容を、助詞や「的」を削って1つの連結に縮めるメモの省略記法は、見出しと表の中では許されますが、地の文では削った語を補って書きます。型として定着した複合語（§2）は、この対象ではありません。例: 「スキルを読んでいない状態での編集」を「スキル未読の編集」と書いてしまう。
- **条件形の直訳**: 日本語の条件形（「〜なら」「〜なければ」）は、行為を分岐させる主張として読まれます。英語の if/when 節にある、主張の場面を設定するだけの用法を条件形に直訳すると、限定としては情報がないまま、裏の場合への含みだけが生じます。場面を設定するだけなら「〜ときも」の形で書くか、場合ごとに文を分けます。例: 「定着した語がなければ造語しない」と条件形で書いてしまう。
- **体言止め**: 完結した文で書き、体言止めを使いません。見出し、表、コミットメッセージの件名のように体言止めが慣習になっている場所は、この規則の対象外です。

### 5. 修辞

- **比喩**: 比喩も §1 の判定に従い、日本語に慣用として定着しているものに限ります。自作の比喩は、そのまま書けば済む内容の飾りにすぎません。定着した比喩でも、近くで重なると読み手の注意を本題から奪います。理解を助けていない比喩は、比喩を使わずに言い直します。
- **直訳の比喩**: 比喩は直訳からも入り込みます（§3）。英語の定型表現には比喩に由来するものが多く、直訳すると、気の利いた比喩を狙った文のように見えます。例: what does it buy you を「〜は何を買うのか」と直訳してしまう。
- **否定を先に置く対比**: 「X ではなく Y」は使える型ですが、読み手は本題の Y より先に、否定される X を読まされます。多用を避け、肯定形で足りるなら肯定形で書きます。
- **飾りの鉤括弧**: 語をそれらしく見せるためだけに鉤括弧で囲むべきではありません。引用、定義、語そのものへの言及ではない鉤括弧は、語を替えるべき合図です。

### 6. 記号

- **コロン**: ラベルと本文を1行に収める形や箇条書きの前置きでは、日本語でも使います（このスキルの「例: 」もその形です）。避けるのは英語の地の文の用法で、段落の中で文の続きをコロンでつなぐもの（「原因は3つ: まず…」）です。文で受けるか、箇条書きに切り出します。
- **ダッシュ**: 「—」「―」「――」のいずれも使いません。文中に挿入句を挟む用法は、読点か括弧で書くか、文を分けます。箇条書きの項目でラベルと説明をつなぐ用法（「ラベル — 説明」）は英語の組版慣習の輸入で、コロンの形（「ラベル: 説明」）か、「は」で受けた文に書き直します。

### 7. 適用の加減

ここまでの規則が退けるのは、実際の用例を持たない語と言い回しであって、比喩や漢語といった特定のカテゴリではありません。種類を一律に禁じると、日本語話者が実際に書く形からかえって離れます。現に、なじみの薄い漢語は避けるべきである一方で、「推す」を「推奨する」に直すように、修正が漢語へ向かう場面も多くあります。判定するのは常に、実際にそう使われているかどうかです。定着してさえいれば、無生物主語（「表が示す」）も、比喩の単発の使用（「骨格」）も、分野の定訳やカタカナ語（「トレードオフ」）も使えます。

これに加えて、プロジェクトの文書で定義され一貫して使われている術語と、ユーザー自身が指示や会話の中で使っている語は、一般の文章で読んだ覚えがなくても使えます。プロジェクトでの定着は、証拠の強さで判断します。ユーザー自身の言葉遣いは強い証拠です。一方、既存文書に現れるというだけでは証拠になりません。既存文書自体が LLM の出力で、ここに挙げた失敗を含んでいることがあるからです。明示的な定義か、ユーザー自身の使用が確かめられたときに、初めて定着とみなします（定義箇所のない語を広めない、という fal-write-docs §4 の規則と同じ判断です）。

推敲・監査の手順と、規則に対応する詳細な用例集は、対になるスキル fal-improve-ja にあります。
