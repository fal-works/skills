# fal-write-ja 用例集

fal-write-ja 本文の規則に対応する、詳細な用例集です。推敲・監査の前に、本文と合わせて通読してパターンを把握します。章立ては本文の節構成とは別に、監査で失敗を見つけるときの手がかりに合わせて二部に分けています。語そのものが手がかりになる失敗は「誤用されやすい語の一覧」に、形の特徴が手がかりになる失敗は「形で見つける失敗」にあり、見出しと項目の（§N）は本文の対応する節を指します。用例はどれも実例であって、規則の範囲を定めるものではありません。ここに載っていない語や形についても、判定は fal-write-ja 本文の基準（実際にそう使われているかどうか）で行います。

## 誤用されやすい語の一覧

LLMの書く日本語で誤用されやすい語の一覧です。各項目の先頭の語は、対象の文章から機械的に探せます。ただし、これは禁止語の一覧ではありません。どの語も、合う場面でならそのまま使えますから、見つけた語は、項目の説明で場面を確かめてから言い直します。閉じた一覧でもありません。誤用されうる語がここに揃っているわけではないので、照合を終えても、語ごとの用例の確認（fal-improve-ja の手順）は省略できません。通読の目的も、個々の語を覚えることではなく、判定のパターンをつかむことにあります。

各項目は、先頭の語の下に、誤用が起きる場面、失敗例とその診断、推奨する言い直しを並べます。失敗例は代表例であり、同じ語の誤用は別の形でも現れます。推奨例は、語の置き換え先ではなく、失敗例の文の言い直しです。

### 場面の不一致（§1）

- 「評定」
  - 場面: レビューの結果など、良し悪しの判断を述べるとき。
  - 失敗例: 「レビューの評定」。評定は成績・査定の場面の語です。
  - 推奨例: 「レビューの評価」。
- 「去就」「存廃」
  - 場面: 機能や関数などを残すか廃止するかを論じるとき。
  - 失敗例: 「機能の去就」「関数の存廃」。「去就」は人の進退の語です。「存廃」は実在する語ですが、関数一つの扱いに使うと大袈裟です。対象との釣り合いも場面のうちです。
  - 推奨例: 「廃止の是非」、あるいは素朴に「残すか廃止するか」。
  - 補足: 「存廃」は制度・路線・組織のような社会的な規模の対象に使うのなら自然です。
- 「較正」
  - 場面: 判断や見積もりの目安を整えることを言うとき。
  - 失敗例: 「判断の較正」。較正（英語なら calibration）は計器や機械学習の語です。
  - 推奨例: 「調整」「加減」。
- 「著述」
  - 場面: モデルなどを書き表す作業を言うとき。
  - 失敗例: 「モデルの著述」。著述（英語なら authoring）は書物の場面の語です。
  - 推奨例: 「モデルの記述」「モデルの書き方」。
- 「経路」
  - 場面: 判断や検討がどう誤ったかを述べるとき。
  - 失敗例: 「判断が失敗した経路」。「経路」は移動や通信の道筋の語で、誤りの成立過程を指す用法はありません。意味は通るので例文は作れますが、実際にそう書かれることはありません。
  - 推奨例: 「どこで判断を誤ったか」のように、内容をそのまま書きます。

### 文体の傾き（§1）

- 「推す」「最強」
  - 場面: 方式や案を勧めたり高く評価したりするとき。
  - 失敗例: 「この方式を推す」「最強の方式」。俗語です。
  - 推奨例: 「推奨する」「最も優れた方式」。
- 「効く」
  - 場面: 設計や対策などの有効さを言うとき。
  - 失敗例: 「この設計が効く」。口語的であり、何に対してどんな効果があるのかを言わないまま、有効そうな印象だけを与える言い回しです。
  - 推奨例: 効果の内容を書くのが理想ですが、口語を消すため「効果的である」「有効である」に置き換えるだけでもよいです。
  - 補足: 「薬が効く」「最適化が効く」のように対象と効果が特定できる組み合わせは、自然な用法です。
- 「通る」
  - 場面: 文書などがある役割を果たせると言うとき。
  - 失敗例: 「用語集として通る」。この「通る」は話し言葉です。
  - 推奨例: 「用語集として通用する」。
- 「割れる」
  - 場面: 逸脱や問題が検査で見つかることを言うとき。
  - 失敗例: 「逸脱が lint で割れる」。「身元が割れる」という俗語寄りの言い回しの転用です。
  - 推奨例: 「lint で検出できる」。
- 「全単射」
  - 場面: A と B が一対一に対応することを言うとき。
  - 失敗例: 「A と B の全単射」。数学的な性質が主題ではない文で使うと、学術文体へ傾きます。
  - 推奨例: 「A と B が過不足なく対応する」。
- 「規律する」
  - 場面: 規約や規則が何かを定めることを言うとき。
  - 失敗例: 「命名規約がモジュール構成を規律する」。一般的な技術文書にしては表現が固すぎます。
  - 推奨例: 「定める」。
  - 補足: 「規律する」は法律・法学の文章でなら使います（「本条が規律する事項」など）。

### 組み合わせの崩れ（§1）

- 「尽きる」
  - 場面: 一覧がすべてを挙げてはいないと断るとき。
  - 失敗例: 「リストは尽きない」。英語の the list is not exhaustive なら自然ですが、その不適切な直訳です。
  - 推奨例: 「この一覧は網羅的ではない」または「すべてを挙げることはできない」など。
  - 補足: 日本語で「尽きる」の主語になるのは、話題や資金のような減っていくものです。
- 「助ける」
  - 場面: ある記述が誤った判断を招きやすいことを言うとき。
  - 失敗例: 「この項目が誤用を助ける」。「助ける」は人や活動を対象にする語で、「誤用」との組み合わせでは使われません。
  - 推奨例: 「誤用を誘発する」「誤用を招く」。

### 英語の語義・言い回しの直訳（§3）

- 「無料」
  - 場面: 英語の for free にあたる内容を書くとき。
  - 失敗例: 「整合性検査が無料で付いてくる」。直訳で、「無料」は金銭の語です。
  - 推奨例: 比喩の置き換え先を探すのではなく、「追加の実装なしに整合性を検査できる」のように、実際の利点をそのまま書きます。
- 「正直」
  - 場面: 英語の honest（honest comparison など）にあたる内容を書くとき。
  - 失敗例: 「正直な比較」。直訳です。この honest はたいてい、自分の案に不利な点も隠さず挙げるという書き手の態度の表明です。強いて言えば「率直な」が近いですが、honest のように慣用句として反射的に添えるものではありません。
  - 推奨例: 率直さの表明が必要な場面でないのなら、honest にあたる表現は省きます。
  - 補足: 比較の条件が対等だという意味なら「公正な比較」などが適しますが、これもやはり公正さを強調したい場面に限られます。
- 「緊張」
  - 場面: 英語の tension（規則や要求どうしの対立）にあたる内容を書くとき。
  - 失敗例: 「スキルの内部に緊張がある」。直訳です。日本語の「緊張」は人や情勢の語で、規則や記述の矛盾には使いません。
  - 推奨例: 「規則どうしが食い違っている」のように、対立の内容をそのまま書きます。
- 「〜と読む」
  - 場面: 英語の read A as B にあたる内容を書くとき。
  - 失敗例: 「省略は既定値と読む」。直訳です。
  - 推奨例: 「省略は既定値とみなす」。
- 「〜に解決する」「〜に畳む」
  - 場面: 英語の resolve や fold にあたる内容を書くとき。
  - 失敗例: 「〜に解決する」「〜に畳む」。直訳です。
  - 推奨例: 本来の技術的な用法の外では使わず、実際に起きることをそのまま書きます（「〜を指す」「〜に含める」）。
- 「漏れる」
  - 場面: 英語の leak にあたる内容を書くとき。
  - 失敗例: 「内部状態が漏れる」「責務が漏れる」。直訳です。英語の leak は、物理的な漏出の像からかなり脱比喩化しており、制御境界の失敗一般を指す上位語として働きます。一方、日本語の「漏れる」は leak よりも物理的な像を喚起しやすく、生きた比喩として目立ちやすいです（「メモリが漏れる」は不自然です）。
  - 推奨例: 「内部状態が外部APIに露出する」「責務の境界が曖昧になる」「内部実装側の制約が利用側に表面化する」など、失敗の種類に応じて語を分け、何がどう破綻しているのかを書きます。
  - 補足: 「確認漏れ」「メモリリーク」「漏れのある抽象化」のように定着した形はそのまま使います。
- 「席」
  - 場面: 英語の earns its place にあたる内容を書くとき。
  - 失敗例: 「この構文が席を得る」。fal-write-docs §1 (Write only what earns its place) の表現からの直訳です。
  - 推奨例: 「この構文は残すに値する」。
  - 補足: 英語で書かれたスキルや指示を読みながら日本語を書くと、この種の直訳が紛れ込みやすいです（fal-write-ja 本文 §3）。
- 「散文」
  - 場面: 英語の prose にあたる内容を書くとき。
  - 失敗例: 「意味は散文が担う」。直訳です。日本語の散文は韻文と対比する語で、図や構造化データとの対比には使いません。
  - 推奨例: 何と対比するかに応じて「地の文」「本文」「自然言語による説明」と書き分けます。
- 「覆う」
  - 場面: 英語の cover にあたる内容を書くとき。
  - 失敗例: 「全ケースを覆う」。直訳です。英語の cover は網羅・対応・補填・援護など広い意味を持ちますが、「覆う」の意味範囲ははるかに狭いです。
  - 推奨例: 「全ケースをカバーする」。「カバーする」はこの広い意味で定着しているカタカナ語です。文脈によっては「網羅する」とも書けます。
  - 補足: 日本語で「覆う」と言えるのは、物理的にかぶさるもの（「雪が畑を覆う」）や、その像を保った比喩（「重い空気が会場を覆う」）などの場合です。
- 「〜の側に置く」
  - 場面: 英語の place the default にあたる内容を書くとき。
  - 失敗例: 「既定を拒否の側に置く」。直訳です。この言い回しに直接対応する日本語はありません。
  - 推奨例: 置き換え先を探すのではなく、言いたかった内容を書きます（「指定がなければ拒否する」「デフォルトでは拒否する」）。
  - 補足: 「既定」は「既定値」「既定の動作」のように連結や修飾の形で使うことが多いです。単独の名詞として言うなら「〜をデフォルトとする」のほうが伝わります。
- 「死ぬ」
  - 場面: 文書などが実態に合わなくなることを言うとき。
  - 失敗例: 「文書が死ぬ」。the document dies の直訳です。
  - 推奨例: 「文書が陳腐化する」、あるいは素朴に「実態に合わない古い文書が残る」のように問題の内容を書きます。
- 「買う」
  - 場面: 英語の what does it buy you にあたる内容を書くとき。
  - 失敗例: 「この制約は何を買うのか」。直訳です。採用の利得を buy と表現するのは英語の定型です。
  - 推奨例: 「この制約から何が得られるのか」。

## 形で見つける失敗

個々の失敗が同じ語の形で繰り返し現れるとは限らず、語の照合では見つけられない失敗です。読んだ覚えのない連結、日本語の文中の英単語、条件形、体言止め、ダッシュといった形の特徴を手がかりに見つけます。用例の一部には「〜とき」の場面の行を残しています。特に圧縮の失敗では、場面の行が圧縮前の正しい言い方を含んでおり、言い直すときの手本になります。

### 圧縮した連結（§2）

検出の手がかりは、読んだ覚えのない複合語・連結です。見つけたら、fal-write-ja 本文 §2 の基準（定着した型に当たるか、省いた語を補うと戻し方が一通りに決まるか）で判定します。以下は造語の類型ごとの実例で、一つの連結が複数の類型に当たることも多いです。

#### 概念への命名

連結が概念の名前として働く類型で、繰り返し使う前提の語や、鉤括弧を付けたくなる語がこれに当たります。

- single source of truth を日本語で言いたいとき
  - 失敗例: 「真実源」。直訳を概念の名前として導入した造語です。
  - 推奨例: 「唯一の情報源」など、あるいは原語のまま「SSoT」と書きます。
- 「何も主張しない」という内容を短く言いたいとき
  - 失敗例: 「無主張」。文の内容を概念の名前に圧縮した造語です。
  - 推奨例: 「主張なし」、または「何も主張しない」と文で書きます。

#### 定着した語の見落とし

同じ意味の定着した語があるなら、型に当たる連結でも造らず、定着した側を使います。

- まだ決まっていない論点を挙げるとき
  - 失敗例: 「未決点」。「変更点」「相違点」のように「○○点」の型はありますが、同じ意味で定着した「未決事項」があります。
  - 推奨例: 「未決事項」「未決定の論点」。
  - 補足: 「未決」という語自体は「未決書類」のように定着した使い方もあります。

#### 助詞や活用を削る圧縮

「～的な」「～された」「～への」などを削って、語を直接つなぐ類型です。削った結果、別の意味に変わることがあります。

- 「決定論的なパース」を短く書きたいとき
  - 失敗例: 「決定論パース」。「的な」を削ると、残るのは主義の名の「決定論」であり、元の意味を保ちません。
  - 推奨例: 「決定論的なパース」のまま書きます。
  - 補足: 同じ失敗の例に「決定論抽出」があります。
- 「確立された記法」を短く書きたいとき
  - 失敗例: 「確立記法」。「された」を削った圧縮です。「確立○○」という型は定着していません。
  - 推奨例: 「確立された記法」「定着した記法」。
- 「本人への確認」を短く書きたいとき
  - 失敗例: 「本人確認」。削った結果が、身元確認という別の意味で定着した語と同形になります。
  - 推奨例: 「本人への確認」のまま書きます。

#### 数詞の連結

- 「主要な2つの懸念」を短く書きたいとき
  - 失敗例: 「主要2懸念」。数詞まで連結に取り込んだ圧縮です。
  - 推奨例: 「主要な2つの懸念」のまま書きます。
  - 補足: 数詞が直接つながるのは、助数詞と、「2ファイル」のようにほぼ助数詞として使える名詞に限られます。

#### 語どうしの関係の消失

- 3段階で移行し、各段階からロールバックできる設計を短く言いたいとき
  - 失敗例: 「3段階ロールバック可能移行」。連ねた語の間の関係が削られ、「3段階」が移行とロールバックのどちらに掛かるのか復元できません。
  - 推奨例: 素朴に「3段階で移行し、各段階からロールバックできるようにする」などと書きます。
  - 補足: 定着した複合語は1語と読まれます。「名前空間分割」は実質2語の連結で、「名前空間の分割」に戻す形が一通りに決まるので、「名前空間分割」のように連結しても構いません。

#### 型の向きの取り違え

- 名前空間をまたぐ参照を短く言いたいとき
  - 失敗例: 「跨ぎ参照」。定着している型は「年跨ぎ」のような「○○跨ぎ」であり、「跨ぎ○○」と前に置く型はありません。
  - 推奨例: 「名前空間をまたぐ参照」と語を補って書きます。

#### 複合語の中の失敗

ここまでの類型が連結の形を分類するのに対し、この失敗はどの形の連結にも重なります。構成する語にも §1〜§3 が適用され、単独の語なら気づける失敗が、連結の中では見えにくくなります。

- 英語の authoring friction にあたる内容を書くとき
  - 失敗例: 「著述摩擦」。直訳で、構成する2語の両方に問題があります。手間・抵抗の意味は英語の friction にはありますが、日本語の「摩擦」（物理現象、集団間の軋轢）にはありません（§3）。「著述」の流用は、語の一覧の「著述」の項と同じです。
  - 推奨例: 「記述の手間」など、内容をそのまま書きます。
- 定義を述べる文章を指すとき
  - 失敗例: 「定義散文」。「散文」が prose の直訳です（語の一覧の「散文」の項）。
  - 推奨例: 「定義の文章」「定義段落」。

#### 使ってよい連結

型として定着した連結は、その連結を読んだ覚えがなくても使えます（fal-write-ja 本文 §2）。

- 接辞のように任意の語と結びつく要素との連結: 「未検証」「未測定」「実運用」「実環境」「検証済み」「定義済み」「保存時」「実行時」「変更不可」「既存機能」「要確認」。
- 後項が種類を表す語で、前項がその内容を限定する2語の連結: 「検証手順」「定義段落」「比較実験」「設計課題」。

型の定着を裏付けるのは、その型でできた既存の語です。

- 「検証手順」: 同じ型に「作業手順」「導入手順」「移行手順」があります。
- 「版差」: 「個人差」「時間差」と同じ「○○差」の型に見えるものの、この形は個々の語が定着しているだけで、新しい語を作る型としては使われていません。この連結は造らず、「バージョン間の差異」のように語を補って書きます。

### 日本語の文中の英単語（§1）

検出の手がかりは、日本語の文中の英単語そのものです。

- 場面: 英語の言い回しを日本語の文中でそのまま使いたくなったとき。
- 失敗例: 「login flow を整理する」。日本語の文中の英単語は術語として読まれますが、login flow は術語ではなく、読者は定義にたどり着けません。
- 推奨例: 「ログイン処理の流れを整理する」と日本語で書きます。
- 補足: コード上の識別子、定着した技術用語、プロジェクトで定義済みの語、ユーザー自身が会話で使った語は、英単語のまま使えます。

### 文と句の組み立て（§4）

- 無生物主語と人の述語
  - 失敗例: 「エラーが沈黙する」。エラーを出さずに失敗することを、人が黙る意味の「沈黙する」で書いています。問題は無生物主語そのものではなく、人の行為の述語と組み合わせた点にあります。
  - 推奨例: 「エラーを出さずに失敗する」。起きることをそのまま書きます。
  - 補足: 無生物主語そのものは日本語でも使います（「規則が定める」「変更が影響する」）。「設定が語る」「関数が名指す」のように、人の動作の述語と組み合わせたときに翻訳調になります。
- 助詞や「的」を削る省略記法
  - 失敗例: 「スキル未読の編集」。「スキルを読んでいない状態で」を名詞の連結に縮めており、地の文では語の関係が読み取りにくいです。
  - 推奨例: 「スキルを読んでいない状態での編集」。削った語を補って書きます。
  - 補足: 見出しや表では、この省略記法を使えます。型として定着した複合語（§2）も、この対象ではありません。
- 条件と帰結を読点でつなぐメモの記法
  - 失敗例: 「使われていない、ならば削除」。条件と帰結を読点でつないだメモの記法で、地の文の形ではありません。
  - 推奨例: 「使われていなければ削除する」。条件形と述語で書きます。
- 条件形の直訳
  - 場面: 主張の場面を設定するとき。
  - 失敗例: 「定着した語がなければ造語しない」。"When no established term exists, don't mint one." の直訳です。造語が問題になる唯一の場面が条件に置かれ、限定としては情報がないまま、「語があれば造語してよい」という逆向きの含みだけが生じます。
  - 推奨例: 「定着した語がないときも、造語はしない」のように「〜ときも」の形で書くか、場合ごとに文を分けます。
  - 補足: 条件が本当に行為を分岐させる文（「エラーが恒久的なら再試行しない」）では、条件形で構いません。
- 体言止め
  - 失敗例: 「この方式を採用」。地の文では述語を欠いた文になります。
  - 推奨例: 「この方式を採用する」。完結した文で書きます。
  - 補足: 見出しや表などで名詞句だけを書くのは許容されます。コミットメッセージや変更履歴のような場面でも、体言止めが慣習ですので問題ありません。

### 記号（§6）

- ラベルと説明をつなぐダッシュ
  - 失敗例: 「「検証手順」 — 「作業手順」「導入手順」が挙がるので、作ってよい。」。ラベルと説明をダッシュでつなぐのは英語の組版慣習です。コロンのラベルの形が日本語でも使えるため、同じ見た目の並びとして紛れ込みやすいです。
  - 推奨例: コロンでつなぐか、「「検証手順」は、…作ってよい。」のように「は」で受けた文にします。
- 挿入句を挟むダッシュ
  - 失敗例: 「…を推奨する場合 — 役割なしでは区別できない場合 — は、…」。挿入句をダッシュで挟むのは英語の用法で、日本語の約物にはありません。
  - 推奨例: 読点か括弧で書くか、文を分けます。
